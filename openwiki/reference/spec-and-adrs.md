## Architecture Decision Records (ADRs)

Key design choices are documented in `/spec/adr/`:

### [ADR-0001: App-only client-credentials authentication](/spec/adr/0001-app-only-client-credentials-auth.md)

- **Decision:** authenticate as an application using OAuth 2.0 client-credentials flow (app-only).
- **Why:** unattended operation, deterministic startup, one-time setup, works identically for stdio and HTTP/SSE.
- **Tradeoff:** no per-user audit trail — every action is attributed to the application principal.
- **Consequences:** server owns token lifecycle (acquire / cache in memory / refresh on expiry).

See [Authentication & OAuth](../architecture/auth.md) for implementation details.

### [ADR-0002: Permission scope — Sites.Selected vs. Sites.ReadWrite.All](/spec/adr/0002-permission-scope-sites-selected.md)

- **Decision:** prefer `Sites.Selected` (scoped to specific sites) over `Sites.ReadWrite.All` (broader).
- **Why:** principle of least privilege; `Sites.ReadWrite.All` grants access to all sites in the tenant.
- **Tradeoff:** `Sites.Selected` requires site-specific admin consent setup.
- **Current:** initial implementations may use `Sites.ReadWrite.All` for simplicity, then tighten to `Sites.Selected` later.

### [ADR-0003: FastMCP framework](/spec/adr/0003-fastmcp-framework.md)

- **Decision:** build the MCP server with **FastMCP** (Python), not other frameworks.
- **Why:** built-in support for both stdio and HTTP/SSE transports, active maintenance, simpler code.
- **Alternatives considered:** mcp-python (lower-level), other Python frameworks.

### [ADR-0004: Microsoft Graph API v1.0](/spec/adr/0004-microsoft-graph-api.md)

- **Decision:** use Microsoft Graph REST API v1.0, not SharePoint REST API.
- **Why:** modern, broadly supported, covers the required operations (read, list, update metadata).
- **When to use SharePoint REST:** managed metadata (taxonomy) columns may require SharePoint REST for term resolution (deferred).

### [ADR-0005: Dual transport — stdio + HTTP/SSE](/spec/adr/0005-dual-transport-stdio-http.md)

- **Decision:** support both stdio (for local use with Claude Desktop / Claude Code) and HTTP/SSE (for remote / shared use).
- **Why:** local development convenience + remote deployment flexibility.
- **Implementation:** FastMCP and configuration (settings.py) support both; transport selection lives in main() only.
- **Current:** stdio is fully working and primary. HTTP/SSE support is designed in but not yet deployed.
